## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-05-25 - [TT Lookup Hash Folding Optimization]
**Learning:** In Transposition Table index calculation (`src/Chess/Engine/TT.hs`), the expression `let k1 = fromIntegral key :: Int; k2 = fromIntegral (key `shiftR` 32) :: Int; idx = ((k1 `xor` k2) .&. mask) * 2` resulted in two `fromIntegral` conversions (which invoke GHC primops `word64ToWord#` then `word2Int#` and `xorI#`). Changing this to `let idx = (fromIntegral (key `xor` (key `shiftR` 32)) .&. mask) * 2` fuses the `xor` to occur natively on `Word64` before the downcast, compiling into a tighter core representation (`GHC.Prim.word2Int# (GHC.Prim.xor# ...)` instead of `GHC.Prim.xorI# (GHC.Prim.word2Int# ...)`) and substantially speeding up TT probe and store operations in microbenchmarks.
**Action:** When performing bitwise arithmetic to fold larger types (like `Word64`) into smaller array index types (`Int`), perform the bitwise operations (`xor`, `shift`) on the original unboxed type first, then apply `fromIntegral` on the final result, rather than casting the operands first. This helps GHC elide intermediate conversions.
