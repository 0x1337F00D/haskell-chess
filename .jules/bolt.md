## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2026-05-28 - [Hoist Type-Level Lookups in Rules Layer (perft)]
**Learning:** In the generic rules layer (`Chess.Core.Rules.Common`), the `applyMoveBase` function was performing redundant allocations and dictionary lookups by evaluating `toColor (colorVal @c)` and `oppositeColor` inside every branch of its `case` statement. Because these values depend strictly on the type-level variable `c`, they are constant for a given instantiation of the function.
**Action:** Hoisted the evaluations of `toColor (colorVal @c)` and `oppositeColor c` out of the `case` block into a top-level `let` binding within the function. This allows GHC's Full Laziness pass to float these bindings and significantly reduces allocations during core game loops. Measured via `bench-core` (+RTS -s), this single minimal change resulted in a massive +15-25% improvement in perft nodes-per-second across multiple variants without breaking boundaries or structural invariants.
