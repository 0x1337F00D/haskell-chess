## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-05-25 - [Transposition Table Unboxed Probing]
**Learning:** `probeTT` was returning a `Maybe (Move, Int, Depth, TTFlag)`, which caused heap allocations for the `Maybe` wrapper and the 4-tuple on every search node in `alphaBetaBody` and `quiescence`. In hot search loops, this boxing significantly increased GC pressure and reduced nodes per second.
**Action:** Introduced `probeTTFast` which returns an unboxed `Word64` directly from the `Vector`, using `maxBound` as an unboxed cache-miss sentinel (which is mathematically guaranteed not to be a valid `packData` output since score and depth are small). Inlined the unpacking (`unpackData`) directly inside the alpha-beta implementation to completely eliminate Transposition Table tuple boxing and intermediate `Maybe` wrappers. Replaced `probeTT` with `probeTTFast` in `Chess.Engine.Search.AlphaBeta` and `Chess.Engine.Search.Quiescence`.
