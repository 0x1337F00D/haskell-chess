## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-06-21 – [Transposition Table Unboxing Optimization]
**Learning:** Returning a boxed `Maybe (Move, Int, Depth, TTFlag)` from `probeTT` forces heap allocation in the extremely hot search paths (`AlphaBeta` and `Quiescence`). Because TT lookups occur for almost every node, this boxing introduces significant garbage collection overhead.
**Action:** Implemented `probeTTFast` returning a direct unboxed tuple `(Word64, Word64)` for the hash key and packed data, and an `unpackDataFast` function to perform bitwise unpacking into an inlined tuple `(Move, Int, Depth, TTFlag)`. Replaced `probeTT` with `probeTTFast` in `AlphaBeta` and `Quiescence` search, eliminating the `Maybe` allocation entirely by checking `entryKey == hash` at the call site.
**Impact:** Reduced total memory allocations in search by roughly 200MB (from ~12.57B to ~12.30B bytes allocated) and improved search time slightly (from ~3.96s to ~3.78s in KiwiPete Depth 6) by bypassing Maybe construction and unpacking overhead.
