## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-06-05 - [Bitboard single bit lookup optimization]
**Learning:** In hot loops like check evasion move generation or scan loops, we frequently need to extract the index of a single bit from a Bitboard, which was previously using `fromMaybe 0 (lsb bitboard)`. `lsb` returns a `Maybe Int`, which forces an allocation of the `Just` constructor and an extra indirection when pattern matched. For tight loops where we know the bitboard is non-zero (like `attackers` in a single check or bits derived from `scanForward`/`forBitboard`), this allocation creates unnecessary GC pressure.
**Action:** Introduced total variants `lsbTotal` and `msbTotal` which map directly to `countTrailingZeros` and `63 - countLeadingZeros` without boxing the result in a `Maybe`. Updated `forBitboard` and evasion loops in `Core.hs` to use these total functions.
**Impact:** Measurable improvement. NPS on bench-core KiwiPete depth 4 increased from 43,769,106 to 44,418,269 (+1.5%). Nodes evaluated faster due to reduced allocation.
