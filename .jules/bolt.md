## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-06-27 – [Transposition Table Unboxing Optimization]
**Learning:** In the chess engine's Transposition Table implementation (`src/Chess/Engine/TT.hs`), `probeTT` was returning a `Maybe (Move, Int, Depth, TTFlag)` tuple. This caused implicit heap allocations (boxing) during extremely hot TT lookups in the alpha-beta and quiescence search loops, creating significant garbage collection pressure.
**Action:** Replaced `probeTT` with `probeTTFast`, which returns an inlined standard unboxed tuple `(Word64, Word64)` of the exact TT entry key and packed data from the vector. The caller now manually compares `entryKey == hash` to detect cache hits before manually unpacking the data via `unpackData`. This completely eliminates `Maybe` constructor and tuple boxing allocations in the search hot path.
**Impact:** Total allocations dropped from ~12.5GB to ~12.2GB, and NPS improved (time for KiwiPete depth 6 decreased from ~4.31s to ~4.08s).
