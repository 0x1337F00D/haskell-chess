## 2024-05-24 - Transposition Table Access Refactor (probeTT/storeTT)
**Learning:** Returning a `Maybe (Move, Int, Depth, TTFlag)` in a hot function like `probeTT` causes heap allocation and significant GC overhead. Also, calling `unpackData` before testing replacement conditions in `storeTT` forced unboxing even when not necessary, leading to wasted allocations. Additionally, extracting `probeTTFast` to return an unboxed `Word64` alongside manually inlining unpacking allows GHC to fuse operations better.
**Action:** Always use a pure, unboxed `Word64` returning function (e.g., `probeTTFast`) paired with `0` as a sentinel for `Nothing` in performance-critical TT reads to eliminate tuples and `Maybe` wrappers in the TT lookup loop where `0` is mathematically guaranteed not to be a valid `packData` output. Extract just the needed bits for replacement checks in `storeTT` to avoid instantiating unboxed wrappers until replacement is actually confirmed.
