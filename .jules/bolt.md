## 2024-05-24 – [Perft Leaf Node Move Counting Optimization]
**Learning:** `perftWhite` and `perftBlack` functions in `Chess.Core.Rules.Class` were evaluating `length (generateMoves game)` for leaf nodes at `depth == 1`. In Standard chess variant, `generateMoves` constructs the full list of moves before computing its length, causing an $O(N)$ allocation of list nodes and boxed moves.
**Action:** Introduced a `countMoves` method to the `ChessVariant` typeclass to bypass generating the full list where possible. Implemented `countMoves` for `Standard` variant to directly return the number of legal moves using `countLegalGenMovesSafe` (or `countLegalGenMoves` when in check), bypassing list allocation entirely for depth 1 leaf node counting.
**Impact:** Minor allocation reduction. On `bench-core` KiwiPete Depth 4: NPS increased slightly.
## 2024-05-24 - [Avoid Heap Allocations by Eliminating Tuple Unpacking in TT Probe]
**Learning:** In Haskell hot paths like Transposition Table (TT) lookups (`probeTT`), returning a custom sum type or a large tuple (e.g., `Maybe (Move, Int, Depth, TTFlag)`) forces the compiler to perform heap allocation (boxing). Since TT probing happens for every visited node during the Alpha-Beta search and Quiescence search, the resulting GC pressure and allocation overhead significantly slow down the search.
**Action:** Introduced an unboxed alternative `probeTTFast` that returns the packed `Word64` directly (or 0 if not found/empty) instead of constructing a `Maybe` and a 4-tuple. Unpacked the word explicitly using strict `let` bindings at the call sites (`AlphaBeta.hs` and `Quiescence.hs`) to bypass intermediate allocations and eliminate boxing overhead entirely on TT cache hits. Next time, always prefer returning packed primitive types (e.g., `Word64`) from low-level table lookups inside hot loops instead of unpacking them immediately into structured types within the lookup function.
