# Architectural Optimization: Explicit Capture Witness in Move GADTs

## 1. The Problem: The Knowledge Gap & Weak Invariants

The current `Move` architecture suffers from two related issues:

1.  **Information Loss**: The Move Generation phase (`MoveGen`) knows exactly which piece is being captured (if any) to validate the move. However, this information is discarded when constructing the `StandardMove` GADT (`StandardMove :: Square -> Square -> PieceType -> Move`).
2.  **Redundant Discovery**: Because the `Move` type lacks capture information, the `applyMove` function must "rediscover" the captured piece by performing `pieceAt` bitboard lookups. This converts what should be an O(1) update into an operation dependent on board state queries.
3.  **Weak Invariants**: The current design permits valid Haskell values that represent impossible chess states (e.g., a "quiet" move to an occupied square, or a "capture" move where the user forgot to handle the victim). The "capture" status is implicit in the board state, not explicit in the `Move` type.

## 2. The Solution: Split Move Constructors

We propose refactoring the `Move` GADT in `Chess.Core.Move.Internal` to make the distinction between "Quiet" and "Capture" moves structural.

Instead of a monolithic `StandardMove` with a `Maybe PieceType` field (which permits the "Nothing, but occupied" invalid state), we split the constructors:

```haskell
data Move (c :: Color) where
  -- | A move to a target square guaranteed to be empty.
  QuietMove ::
    { qmFrom :: Square
    , qmTo   :: Square
    , qmMoving :: PieceType
    } -> Move c

  -- | A move to a target square guaranteed to be occupied by an enemy.
  -- Carries the proof of the victim's identity.
  CaptureMove ::
    { cmFrom :: Square
    , cmTo   :: Square
    , cmMoving :: PieceType
    , cmCaptured :: PieceType -- ^ The explicit witness of the captured piece.
    } -> Move c

  -- | Promotion without capture.
  PromotionMove ::
    { pmFrom :: Square
    , pmTo   :: Square
    , pmPromotion :: PieceType
    } -> Move c

  -- | Promotion with capture.
  PromotionCaptureMove ::
    { pcmFrom :: Square
    , pcmTo   :: Square
    , pcmPromotion :: PieceType
    , pcmCaptured :: PieceType
    } -> Move c

  -- (Castling, EnPassant, Drop, Castling960 remain unchanged)
```

## 3. Why It Improves Speed and Design

### ⚡ Performance (Speed)
*   **O(1) Execution**: `applyMove` (and `applyMoveBase`) no longer needs to branch or lookup the board to decide *if* a piece is captured or *what* it is. The `CaptureMove` constructor provides the exact instructions: "Remove a Bishop from F5".
*   **Reduced Memory Traffic**: By avoiding `pieceAt` (which accesses multiple bitboards), we reduce cache pressure in the engine's tightest loop (search/perft).

### 🧠 Semantic Clarity (Design)
*   **Illegal States Unrepresentable**: It is now impossible to represent a "Capture" without specifying the victim. It is impossible to process a `CaptureMove` and "forget" to handle the captured piece.
*   **Total Functions**: Logic for variants like *Atomic* (explode on capture) or *Crazyhouse* (pocket the captured piece) becomes total. They pattern match on `CaptureMove` and immediately access `cmCaptured`. They do not need to assume the board is consistent with the move; the move *is* the truth.
*   **Parse, Don't Validate**: This aligns with the philosophy that a `Move` is a *proof* generated by a trusted kernel (`MoveGen`), not just a request.

## 4. Trade-offs and Limitations

*   **Refactoring Cost**: This is a breaking change. All pattern matches on `StandardMove` in the codebase (Rule variants, tests) must be updated.
*   **Context-Dependent Parsing**: Constructing a `Move` from a UCI string (e.g., "e2e4") requires access to the `Board` to determine if it should be `QuietMove` or `CaptureMove`. This correctly enforces the boundary between "untrusted text" and "verified moves".

## 5. Location in Architecture

*   **Definition**: `Chess.Core.Move.Internal`
*   **Construction**: `Chess.Core.Rules.Common.toCoreMove` (Trusted Kernel)
*   **Consumption**: `Chess.Core.Rules.*` (Variant logic)
